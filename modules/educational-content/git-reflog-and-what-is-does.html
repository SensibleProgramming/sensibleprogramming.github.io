When and Why to Use Git Reflog

Master the "Flight Data Recorder" of your Git repository and never fear losing a commit again.

1. Overview

git reflog (Reference Log) is a local mechanism that records every time the tip of a branch or the HEAD pointer is updated in your repository. While git log provides a history of the project’s commits, git reflog provides a history of your local actions.

It tracks where your repository’s pointers have been, capturing movements caused by commits, checkouts, resets, merges, and rebases. It essentially serves as a flight data recorder for your local Git environment, allowing you to recover data that might otherwise appear permanently lost.

2. Main Learning Points

Action vs. Commit

Understand the difference between public project history (log) and private local activity (reflog).

Reverse Destruction

Master reversing operations like git reset --hard that appear to delete data.

Data Recovery

Learn to recover deleted branches and "dangling" commits not attached to any named branch.

Navigation

Gain proficiency in reading and navigating the HEAD@{n} syntax for restoration.

3. Detailed Descriptions of Concepts

01 Difference between git log and git reflog

  • Scope: git log is part of the repository's data and is synchronized. git reflog is strictly local.
  • History: git log follows parentage. If you reset, commits disappear from log. git reflog records the movement itself, keeping a chronological record regardless of branch history.

02 Key Use Cases

Accidental git reset --hard

Scenario: You intended a soft reset but performed a hard reset, wiping recent work.

Recovery: Run git reflog. Find the entry before the reset (e.g., HEAD@{2}) and run git reset --hard HEAD@{2}.

Recovering a Deleted Branch

Scenario: You deleted a branch that still had experimental code you needed.

Recovery: Look for the checkout: moving from branch to main entry. Use the commit hash: git checkout -b recovered-branch <hash>.

Undoing a Messy Rebase

Scenario: Interactive rebase went sideways, and history is now broken.

Recovery: Look for rebase (start): checkout <branch>. Reset to that stable state using HEAD@{n}.

Finding "Lost" Commits

In a "detached HEAD" state? If you didn't branch before checking out main, your commits are dangling. Use git reflog to find the hashes and create a branch from them immediately.

Understanding Reflog Syntax

Reflog entries are identified by a selector: name@{qualifier}.

HEAD@{0} Current position of HEAD

HEAD@{1} Position one move ago

HEAD@{"1 hour ago"} Position 60 mins ago

main@{2} Main pointer two changes ago

Note: Reflog is local. If you delete the repository folder, the reflog is gone. It is an undo tool, not a hardware backup.

4. Final Summary

git reflog is the ultimate safety valve for the professional developer. While git log is the story of the code’s evolution, git reflog is the story of your interaction with the tool.

Whenever a command produces an unexpected result—whether it’s a lost commit, a failed rebase, or an accidental reset—the reflog should be your first point of reference. By learning to navigate the reference log, you remove the fear of "breaking" your repository.
