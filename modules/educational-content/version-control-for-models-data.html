01. Overview

In traditional software engineering, versioning source code with Git is sufficient to ensure reproducibility. However, in machine learning, the code is only one part of the equation. A model’s behavior is defined by a combination of the code, the specific version of the dataset used for training, and the environment/hyperparameters.

Traditional Git is not designed to handle large binary files (weights) or massive datasets (TB-scale). MLOps versioning bridges this gap by creating a system where data and models are versioned alongside code, ensuring that any model in production can be traced back to the exact data and parameters that created it.

03. Detailed Descriptions of Concepts

The Limitations of Git for ML

Git was designed for text-based source code. It struggles with ML assets for three primary reasons:

  • File Size: Git stores the entire history of a file. Storing multi-gigabyte datasets or model weights quickly leads to repository bloat.
  • Binary Formats: Data formats like .parquet or model weights .pth are binary; Git cannot show "diffs".
  • Storage Efficiency: Git expects files to live locally, whereas ML workflows require cloud-native storage like S3 or GCS.
Key Concept: Meta-Pointers Instead of storing the data in Git, we store meta-pointers—small text files containing hashes of the large data.

Data Versioning: Principles and Tools

Data versioning treats datasets as immutable snapshots.

DVC

Creates .dvc pointer files. Use git checkout for the code and dvc pull for the corresponding data.

LakeFS

Provides a Git-like interface over an object store (S3). Allows branching and merging directly on the data lake.

Model Versioning: Weights and Metadata

Model versioning involves tracking the "Model Registry." This is more than just saving a pickle file; it's about context.

  • Weights: Serialized parameters (.onnx, .pth).
  • Metadata: Hyperparameters, metrics, and library versions.
  • State: Tagged as Staging, Production, or Archived.

The Concept of Lineage

Lineage is the "audit trail" of an ML model. To achieve full lineage, you must link:

Code Version + Data Version + Environment = Verifiable Artifact

"Which dataset version caused this model to exhibit bias in production?" - Lineage provides the answer.

04. Practical Workflow

1

Initialize Project

Initialize Git and DVC. Configure a remote storage (S3/GCS).

dvc remote add -d mys3 s3://my-bucket/data
2

Version the Data

Track raw data with DVC and commit the pointer to Git.

dvc add data/raw_images.zip && git add raw_images.zip.dvc
3

Experimental Run

Execute training scripts while logging artifacts to MLflow.

4

Link and Commit

Capture the Run ID in your git commit message for ultimate traceability.

05. Final Summary

Versioning in MLOps is the evolution of software versioning to account for the non-deterministic nature of data. While Git manages your logic, tools like DVC and MLflow manage your state and results.

Key Takeaway

Never store large data or models directly in Git. Maintain a persistent link between code, data hash, and experiment metadata.
